# PlateauBreaker 技術架構與擴展指南

## 1. 專案概述

PlateauBreaker 是一個全端 MVP (Minimum Viable Product) 專案，旨在提供個人健康數據分析系統，用於記錄每日數據、視覺化趨勢、偵測體重高原期，並透過規則引擎分析原因，最終生成人類可讀的洞察報告。本專案採用 Vue 3 (前端) + FastAPI (後端) 的技術棧，並以 SQLite 作為輕量級資料庫。

## 2. 核心高原期偵測演算法

高原期偵測是 PlateauBreaker 的核心功能，它結合了兩種規則來判斷用戶是否進入體重高原期：

### 2.1. 數據準備

所有分析均基於用戶最近的健康記錄。系統會首先確定最新的記錄日期 (`latest_date`)，然後從該日期回溯，提取「最近 7 天」(`last7`) 和「前一個 7 天」(`prev7`) 的數據。為了確保分析的有效性，高原期偵測至少需要最近 7 天內有 **5 天** 的有效記錄。

### 2.2. 偵測規則

1.  **規則 A (趨勢變化)**：比較「最近 7 天的平均體重」與「前一個 7 天的平均體重」的變化。如果變化量小於 0.2 公斤，則視為高原期。
    -   `avg_change = last7_avg - prev7_avg`
    -   `rule_a_plateau = abs(avg_change) < 0.2`

2.  **規則 B (波動範圍)**：分析「最近 7 天的體重波動範圍」。如果這 7 天內的體重最大值與最小值之差小於或等於 0.6 公斤（即波動在 ±0.3 公斤內），則視為高原期。
    -   `last7_fluctuation = last7_max - last7_min`
    -   `rule_b_plateau = last7_fluctuation <= 0.6`

### 2.3. 狀態判斷邏輯

系統會綜合兩條規則的結果來判斷最終的體重狀態：

-   **`insufficient_data`**：如果最近 7 天的有效記錄少於 5 天，則數據不足以進行分析。
-   **`plateau`**：如果規則 A 和規則 B 都指示高原期，或者規則 A 處於邊緣狀態且規則 B 指示高原期。
-   **`losing`**：如果體重趨勢明顯下降（`avg_change <= -0.2`）。
-   **`gaining`**：如果體重趨勢明顯上升（`avg_change >= 0.2`）。
-   **`losing` / `gaining` (基於波動)**：如果規則 A 的數據不足，則僅依賴規則 B。若非高原期，則根據最近 7 天的第一天和最後一天的體重差來判斷是「減重中」還是「體重增加」。

### 2.4. 數據完整性標誌

為了提高分析的透明度，如果最近 7 天的數據不完整（少於 7 天但不少於 5 天），系統會在高原期偵測結果中添加一條訊息，指出「分析置信度因記錄不完整而降低」。

## 3. 原因分析與數據權重系統

原因分析模組 (`reason_analyzer.py`) 負責識別導致體重高原期的潛在因素。每個潛在原因都被賦予一個「嚴重性權重」，用於更精確地排序和呈現。

### 3.1. 潛在原因

1.  **`SleepIssue` (睡眠不足)**：平均睡眠時間少於 6 小時。
2.  **`CalorieIssue` (卡路里超標)**：平均卡路里攝入量超過設定的目標值。
3.  **`WeekendOvereating` (週末過食)**：週末平均卡路里攝入量比平日平均高出 20% 以上。
4.  **`ExerciseDrop` (運動量下降)**：最近 7 天的平均運動量少於前一個 7 天的 70%。
5.  **`DataMissing` (數據缺失)**：最近 7 天內有超過 2 天的記錄缺失。

### 3.2. 嚴重性權重 (`SEVERITY_WEIGHTS`)

為了使原因排名更具意義，每個原因類型都有一個標準化的嚴重性權重：

| 原因類型 | 權重 | 說明 |
| :--------------- | :--- | :----------------------------------------------------------------------------------- |
| `SleepIssue` | 1.0 | 睡眠對體重管理至關重要，權重較高。 |
| `CalorieIssue` | 1.0 | 卡路里攝入是體重變化的直接因素，權重較高。 |
| `WeekendOvereating` | 0.8 | 週末習慣對長期趨勢有影響，但不如每日卡路里總量直接。 |
| `ExerciseDrop` | 0.8 | 運動量下降會影響熱量消耗，但其影響可能不如飲食直接。 |
| `DataMissing` | 1.2 | **最高權重**。數據缺失嚴重影響分析準確性，應優先提示用戶。 |

每個原因的最終「嚴重性分數」是根據其具體數據（例如，睡眠不足的小時數、卡路里超出的百分比）乘以其對應的權重計算得出。這確保了排名不僅考慮了問題的程度，也考慮了其對體重高原期的潛在影響力。

### 3.3. 數據驅動的摘要生成

`summary_generator.py` 模組負責將高原期狀態和原因分析結果轉化為人類可讀的摘要。為了提供更具體的洞察，摘要現在會包含關鍵的數值信息：

-   如果存在 `DataMissing`，摘要會優先提示「分析置信度因記錄不完整而降低。」
-   針對 `SleepIssue`，會顯示平均睡眠小時數。
-   針對 `CalorieIssue`，會顯示超出目標的百分比。
-   針對 `DataMissing`，會顯示具體缺失的天數。

## 4. 未來擴展建議

PlateauBreaker MVP 奠定了堅實的基礎，未來可以從以下幾個方向進行擴展：

### 4.1. 用戶認證與多用戶支持

-   **需求**：目前專案無用戶認證，所有數據都儲存在單一 SQLite 資料庫中。為支持多用戶，需要引入用戶註冊、登錄和數據隔離機制。
-   **建議**：
    -   **後端**：整合 OAuth2 (例如：FastAPI 的 `fastapi.security.OAuth2PasswordBearer`) 或 JWT 認證。將 `HealthRecord` 模型與 `User` 模型關聯。
    -   **資料庫**：為每個用戶的數據添加外鍵約束，確保數據隔離。
    -   **前端**：添加登錄/註冊頁面，並在 API 請求中包含認證 Token。

### 4.2. 更豐富的數據輸入與追蹤

-   **需求**：擴展可追蹤的健康指標，提供更全面的健康視圖。
-   **建議**：
    -   **後端**：在 `HealthRecord` 模型中添加更多字段，例如：體脂率、肌肉量、飲水量、心情指數、壓力水平等。
    -   **前端**：更新記錄表單，加入新的輸入字段和對應的視覺化組件。

### 4.3. 進階分析與機器學習

-   **需求**：引入更複雜的分析模型，提供更個性化和預測性的洞察。
-   **建議**：
    -   **機器學習**：使用時間序列分析 (如 ARIMA, Prophet) 預測體重趨勢。利用聚類算法識別用戶的行為模式。引入監督學習模型，根據用戶數據預測高原期發生的可能性。
    -   **個性化推薦**：基於用戶的數據和高原期原因，提供個性化的飲食、運動和生活習慣建議。
    -   **外部 API 整合**：考慮整合第三方數據源，例如：健身追蹤器 (Garmin, Fitbit)、食物數據庫 (Open Food Facts) 等，自動化數據輸入。

### 4.4. 視覺化與用戶體驗優化

-   **需求**：提升數據視覺化的豐富度和用戶界面的互動性。
-   **建議**：
    -   **更多圖表類型**：除了線圖，可以添加柱狀圖、散點圖、熱力圖等，展示不同維度的數據關係。
    -   **互動式報告**：允許用戶自定義報告時間範圍、篩選數據，並生成可分享的報告。
    -   **移動端適配**：進一步優化響應式設計，提供更流暢的移動端體驗。

### 4.5. 通知與提醒系統

-   **需求**：在用戶可能進入高原期或數據缺失時，主動發送通知。
-   **建議**：
    -   **後端**：實現定時任務 (Cron Job)，定期運行分析並發送通知。可以整合郵件服務 (SendGrid) 或消息推送服務 (Firebase Cloud Messaging)。
    -   **前端**：顯示應用內通知，或請求瀏覽器通知權限。

## 5. 結論

PlateauBreaker MVP 是一個功能完備、代碼整潔且邏輯嚴謹的基礎專案。本指南旨在幫助您理解其核心機制，並為未來的擴展提供清晰的方向。透過逐步迭代，PlateauBreaker 有潛力發展成為一個強大而全面的個人健康管理工具。
